## CPU의 분기 예측
---
CPU 구조 상, 분기 예측을 실패했을 때, 기존 파이프라이닝을 멈추고 명령어를 비워(stall)주는 과정이 필요하다. 아니, 명령어를 비운다고 왜? 왜냐하면, 조건식이 평가될 때까지 기다리는 것보다 일단 실행하고 예측이 맞으면 그대로 진행, 아니면 복구하는 게 더 낫기 때문이다. 물론, 실패했을 때 다시 명령어를 가져오고 메모리 상태를 맞춰주는 등 비용이 적지 않다. 그렇기에, 현대적 CPU는 분기 예측이라는 걸 한다. 즉, 분기 예측이 실패한 경로는 점점 시도하지 않고, 성공한 경로는 점점 시도한다.


## C++의 분기 예측 힌트
---
```cpp
[[likely]] if( condition ){
    // do-something
}

[[unlikely]] if( condition ){
	// do-something
}
```

사실은 분기 예측 힌트를 주지 않는다고 해서 CPU가 분기 예측을 안하는 것은 아니다. 애초에 컴파일러가 코드를 어떻게 생성한다고 해서 CPU의 동작을 변경할 수도 없다! 그렇기에 이건 어디까지나 힌트인 것이다.

이를 지정해주면 컴파일러는...
- likely 경로를 좀 더 공격적으로 인라이닝하거나, 레지스터 할당을 유리하게.
- 혹은 코드 레이아웃(branch 바로 아래 배치 vs 주소 점프한 곳에 배치)을 변경.

예컨데 명령어 목록은 어떤 덩어리 단위로 메모리에 올라오기에, 더 자주 실행할 코드를 더 가까이 배치하도록 할 수 있다. 프로그래머가 거짓말을 하더라도 분기 예측은 작동하고 더 자주 실행되는 경로로 더 많이 예측하게 될 테지만(물론 처음 예측은 더 잘 지정해줬을 때보다 더 느릴 것이다.), 프로그램의 레이아웃 자체에 대한 약간의 최적화 여지는 이런 속성을 잘 지정했을 때만 챙겨갈 수 있다.
